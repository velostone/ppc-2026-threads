## 1. Контекст

Решается задача выделения ребер на изображении с использованием оператора Собеля. Математическая постановка и
последовательный алгоритм описаны в файле отчета `seq/report.md`.
Алгоритм хорошо распараллеливается ввиду отсутствия зависимостей по данным между итерациями циклов.
Предлагается организация параллелизма с помощью технологии OpenMP.

## 2. Постановка задачи

На вход алгоритма подается изображение типа `Image`, пиксели которого упакованы в `std::vector<uint8_t>`.
На выходе ожидается результирующий вектор `std::vector<int>`, компоненты которого принимают значение 0,
если ребро отсутствует, и `magnitude`, если ребро есть. За деталями обращаться к файлу `seq/report.md`.

## 3. Базовый алгоритм

Последовательная версия алгоритма в цикле по изображению обходит внутренние пиксели, применяя к ним ядра оператора Собеля
и вычисляя нормы соответствующих градиентов. Решение о наличии ребра принимается в результате сравнения значения нормы
с некоторым заданным порогом чувствительности `threshold`. За деталями обращаться к файлу `seq/report.md`.

## 4. Схема распараллеливания

Внешний вычислительный цикл алгоритма был распараллелен с помощью директивы `#pragma omp parallel for`, что соответствует
распределению работы между потоками по наборам строк входного изображения. Число потоков устанавливается как
`num_threads(ppc::util::GetNumThreads())`, при этом используется переменная окружения `PPC_NUM_THREADS`.
Работа распределяется статическим образом, равномерно и непрерывно, что обеспечивается параметром директивы `schedule(static)`.
Данный вид расписания приводит к наибольшей производительности в данной задаче, поскольку не создает накладных расходов
в виде очереди задач и позволяет равномерно нагрузить все доступные потоки, выполняющие одинаковые по вычислительной
сложности операции. Для улучшения читаемости кода задан ручной режим управления областью видимости переменных (`default(none)`),
разделяемые данные выписаны явно как `shared(kSobelX, kSobelY, input, input_data, output`. Потоки обращаются к общей памяти
для умножения на ядра, чтения пикселей изображения и заполнения результата. Гонка данных при записи при этом не возникает,
поскольку каждый поток пишет в свою часть адресного пространства вектора `output`. Счетчики итераций, вычисляемые компоненты
градиента и его норма относится к `private` переменным. Редукция в рассматриваемом алгоритме не используется.
Средствами OMP обеспечивается неявная синхронизация в конце параллельной области `#pragma omp parallel for`.
Явные барьеры не используются.

## Приложение

```cpp
// File: omp/src/ops_omp.cpp
#pragma omp parallel for num_threads(ppc::util::GetNumThreads()) schedule(static) default(none) \
    shared(kSobelX, kSobelY, input, input_data, output)
for (std::size_t j = 1; j < h_ - 1; ++j) {
  for (std::size_t i = 1; i < w_ - 1; ++i) {
    int gx = 0;
    int gy = 0;

    for (int ky = -1; ky <= 1; ++ky) {
      for (int kx = -1; kx <= 1; ++kx) {
        std::uint8_t pixel = input_data[((j + ky) * w_) + (i + kx)];
        gx += pixel * kSobelX.at(ky + 1).at(kx + 1);
        gy += pixel * kSobelY.at(ky + 1).at(kx + 1);
      }
    }

    auto magnitude = static_cast<int>(round(std::sqrt((gx * gx) + (gy * gy))));
    output[(j * w_) + i] = (magnitude > input.threshold) ? magnitude : 0;
  }
}
```
