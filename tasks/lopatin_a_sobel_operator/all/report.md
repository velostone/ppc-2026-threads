## 1. Контекст

Решается задача выделения ребер на изображении с использованием оператора Собеля. Математическая постановка и
последовательный алгоритм описаны в файле отчета `seq/report.md`.
Алгоритм хорошо распараллеливается ввиду отсутствия зависимостей по данным между итерациями циклов.
Предлагается организация двухуровневого параллелизма с помощью технологий MPI и OpenMP.
Такая реализация может быть полезна для кластерных систем: MPI распределяет данные между узлами, а OpenMP эффективно
использует доступные вычислительные ресурсы каждого узла.

## 2. Постановка задачи

На вход алгоритма подается изображение типа `Image`, пиксели которого упакованы в `std::vector<uint8_t>`.
На выходе ожидается результирующий вектор `std::vector<int>`, компоненты которого принимают значение 0, если ребро отсутствует,
и `magnitude`, если ребро есть. За деталями обращаться к файлу `seq/report.md`.

## 3. Базовый алгоритм

Последовательная версия алгоритма в цикле по изображению обходит внутренние пиксели, применяя к ним ядра оператора Собеля
и вычисляя нормы соответствующих градиентов. Решение о наличии ребра принимается в результате сравнения значения нормы
с некоторым заданным порогом чувствительности `threshold`. За деталями обращаться к файлу `seq/report.md`.

## 4. Межпроцессная схема

Параллельная реализация выполнена с использованием топологии стандартного коммуникатора `MPI_COMM_WORLD`. В качестве
корневого процесса используется процесс с рангом 0. Он выполняет распределение данных, собирает локальные результаты,
обрабатывает хвост.

Параллельный алгоритм включает в себя несколько этапов:

1. Пиксели изображения распределяются равномерно по процессам с размером порции `rows / proc_num`, где `proc_num` - число
процессов коммуникатора `MPI_COMM_WORLD`, `/` - целочисленное деление.
2. Каждый процесс вычисляет свой диапазон итераций алгоритма Собеля и сохраняет результат в локальном буфере `local_output`.
3. Обработкой "хвоста" (при его наличии) занимается процесс с рангом 0 в отдельном цикле.
4. Выполняется аккумуляция локальных результатов путем использования функции `MPI_Gatherv` с предварительно подготовленными
значениями `sendcount`, `recvcounts`, `displs`.
5. Результат применения оператора Собеля рассылается всем процессам с помощью функции `MPI_Bcast` в целях синхронизации
данных для прохождения тестов (тестовая инфраструктура требует наличия результата на всех процессах).

## 5. Внутрипроцессная схема

Внешний вычислительный цикл алгоритма на каждом узле распараллелен с помощью директивы `#pragma omp parallel for`,
что соответствует распределению работы между потоками по наборам строк входного изображения из переданной узлу порции вычислений.
Число потоков устанавливается как `num_threads(ppc::util::GetNumThreads() / proc_num)`, при этом используются переменные
окружения `PPC_NUM_PROC` и `PPC_NUM_THREADS`. Работа распределяется статическим образом, равномерно и непрерывно,
что обеспечивается параметром директивы `schedule(static)`. Данный вид расписания приводит к наибольшей производительности
в данной задаче, поскольку не создает накладных расходов в виде очереди задач и позволяет равномерно нагрузить все
доступные потоки,выполняющие одинаковые по вычислительной сложности операции.
Для улучшения читаемости кода задан ручной режим управления областью видимости переменных (`default(none)`),
разделяемые данные выписаны явно как `shared(kSobelX, kSobelY, img, input_data, output, start, end, shift)`.
Потоки обращаются к локальной памяти для умножения на ядра, чтения пикселей изображения и заполнения результата.
Гонка данных при записи при этом не возникает, поскольку каждый поток пишет в свою часть адресного пространства
вектора `output`. Счетчики итераций, вычисляемые компоненты градиента и его норма относится к `private` переменным.
Редукция в рассматриваемом алгоритме не используется.
Средствами OMP обеспечивается неявная синхронизация в конце параллельной области `#pragma omp parallel for`.
Явные барьеры не используются.

## 6. Детали реализации

- Файлы: all/include/ops_all.hpp, all/src/ops_all.cpp
- Каждый процесс в параллельном режиме выполняет `rows / n_threads` итераций, нулевой процесс дополнительно обрабатывает
хвост размера `rows % n_threads`
- Если число созданных процессов больше числа строк изображения, то избыточные процессы существуют фиктивно,
необмениваясь данными и не выполняя вычислений.
- Процессы обрабатывают свою порцию строк и записывают результат в локальный буфер `local_output`
- Детали реализации непосредственного алгоритма оператора Собеля в его последовательной версии описаны в файле `seq/report.md`

## 10. Выводы

При решении задачи выделения ребер на изображении с помощью оператора Собеля двухуровневый параллелизм, организованный
с помощью технологий MPI и OpenMP, дал ускорение до 1.82 раз на 3 процессах с 2 потоками, что соответствует числу
логических ядер процессора. Наибольшая эффективность 0.39 была достигнута на 2 процессах с 2 потоками.
Дальнейшее масштабирование привело к падению эффективности до 0.17 при 4 процессах с 2 потоками.
По сравнению с OMP, TBB, STL реализациями наблюдалось сильное замедление во всех конфигурациях, что может быть вызвано
накладными расходами коммуникаций `MPI_Gatherv` и `MPI_Bcast`, неэффективной обработкой хвоста после синхронизации
(корневой процесс будет простаивать в ожидании остальных процессов). Следовательно, использование гибридного параллелизма
с MPI избыточно и даже вредно для машин с общей памятью.

## Приложение

```cpp
// File: all/src/ops_all.cpp
int proc_num = 0;
MPI_Comm_size(MPI_COMM_WORLD, &proc_num);
int proc_rank = 0;
MPI_Comm_rank(MPI_COMM_WORLD, &proc_rank);

const auto &input = GetInput();

const auto rows = input.height - 2;
const auto n_procs = std::min(static_cast<std::size_t>(proc_num), rows);

const std::size_t portion = rows / n_procs;

lopatin_a_sobel_operator::OutType local_output(portion * input.width, 0);

if (std::cmp_less(proc_rank, n_procs)) {
  std::size_t start = 1 + (proc_rank * portion);
  std::size_t end = 1 + ((proc_rank + 1) * portion);
  RunSobel(input, start, end, local_output, start);
}

auto &output = GetOutput();

std::vector<int> recvcounts(proc_num, 0);
std::vector<int> displs(proc_num, 0);

for (size_t i = 0; i < n_procs; ++i) {
  recvcounts[i] = static_cast<int>(portion * input.width);
  displs[i] = static_cast<int>((1 + i * portion) * input.width);
}

int sendcount = std::cmp_less(proc_rank, n_procs) ? static_cast<int>(local_output.size()) : 0;

MPI_Gatherv(local_output.data(), sendcount, MPI_INT, output.data(), recvcounts.data(), displs.data(), MPI_INT, 0,
            MPI_COMM_WORLD);

if (proc_rank == 0) {
  const std::size_t tail = rows % n_procs;

  if (tail != 0) {
    std::size_t start = input.height - 1 - tail;
    RunSobel(input, start, input.height - 1, output, 0);
  }
}

MPI_Bcast(output.data(), static_cast<int>(output.size()), MPI_INT, 0, MPI_COMM_WORLD);
```
